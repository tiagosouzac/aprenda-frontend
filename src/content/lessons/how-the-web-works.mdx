  <p>
    Todo site que você abre — Google, YouTube, seu banco — passa pelas mesmas peças invisíveis
    no momento em que você pressiona Enter na barra de endereço. Nomes viram endereços numéricos,
    o navegador conversa com um servidor, bytes atravessam a rede e um documento de texto vira a
    página na sua tela.
  </p>
  <p>
    Entender esse caminho muda como você vai escrever HTML, CSS e JavaScript nos próximos
    módulos. Quando um site quebra, quando carrega devagar, quando algo não aparece — a resposta
    quase sempre mora em uma dessas peças.
  </p>
  <p>
    Este módulo percorre a sequência inteira, do endereço digitado até o pixel pintado.
  </p>

<LessonSection number="01" title="O caminho de uma URL" id="caminho-url">
  <p>
    Quando você digita <code>example.com</code> e pressiona Enter, uma sequência acontece em
    poucos segundos:
  </p>
  <ol>
    <li>O navegador extrai o host da URL e pede ao DNS o IP do servidor.</li>
    <li>Abre uma conexão TCP com esse IP.</li>
    <li>Se a URL for HTTPS, negocia TLS (o cadeado na barra de endereço).</li>
    <li>Envia uma requisição HTTP pedindo o recurso — geralmente a raiz, <code>/</code>.</li>
    <li>O servidor responde com um status, headers e o conteúdo (normalmente HTML).</li>
    <li>O navegador começa a parsear o HTML e baixa os recursos referenciados (CSS, JS, imagens).</li>
    <li>Monta a árvore de renderização, calcula o layout e pinta a página.</li>
  </ol>
  <p>Cada peça dessa sequência aparece nas próximas seções.</p>
</LessonSection>

<LessonSection number="02" title="DNS — do nome ao endereço" id="dns">
  <p>
    Computadores se comunicam usando IPs (endereços numéricos). Pessoas lembram de nomes. O DNS
    (<em>Domain Name System</em>) é a lista telefônica da internet: traduz
    <code>example.com</code> em algo como <code>93.184.216.34</code>.
  </p>
  <p>
    O processo é rápido e invisível. Seu navegador pergunta ao sistema operacional, que consulta
    o servidor DNS configurado (normalmente o do seu provedor). Esse servidor pode já ter a
    resposta em cache, ou perguntar para servidores acima dele até chegar no servidor
    autoritativo do domínio.
  </p>
  <CodeBlock
    lang="bash"
    code={dnsLookup}
    caption="No terminal, `dig` resolve o nome e mostra o IP correspondente."
  />
</LessonSection>

<LessonSection number="03" title="HTTP — a conversa entre cliente e servidor" id="http">
  <p>
    HTTP é o protocolo que define como cliente e servidor conversam. A conversa tem um padrão
    fixo: o cliente faz uma <strong>requisição</strong>; o servidor devolve uma
    <strong>resposta</strong>.
  </p>
  <p>
    A requisição tem um método (o verbo da ação), um caminho (o que se quer), headers (metadados)
    e, às vezes, um corpo com dados:
  </p>
  <CodeBlock
    lang="bash"
    code={httpRequest}
    caption="Uma requisição HTTP pedindo a página inicial de example.com."
  />
  <p>
    A resposta tem um status (como foi), headers e um corpo com o conteúdo pedido:
  </p>
  <CodeBlock
    lang="bash"
    code={httpResponse}
    caption="A resposta do servidor, começando pelo status 200 OK e seguindo com o HTML."
  />
  <p>Os métodos que você vai ver com mais frequência:</p>
  <ul>
    <li><code>GET</code> — buscar um recurso (ler)</li>
    <li><code>POST</code> — enviar dados para criar algo</li>
    <li><code>PUT</code> — substituir um recurso existente</li>
    <li><code>DELETE</code> — remover um recurso</li>
  </ul>
  <p>
    A maior parte do que você abre no navegador é <code>GET</code>. Formulários e APIs usam os
    outros.
  </p>
</LessonSection>

<LessonSection number="04" title="Status codes — o que o servidor está dizendo" id="status-codes">
  <p>
    Toda resposta começa com um status — três dígitos que resumem o que aconteceu. O primeiro
    dígito classifica a categoria:
  </p>
  <ul>
    <li><strong>2xx — sucesso.</strong> <code>200 OK</code>, <code>201 Created</code>.</li>
    <li><strong>3xx — redirecionamento.</strong> <code>301 Moved Permanently</code>, <code>304 Not Modified</code>.</li>
    <li><strong>4xx — erro do cliente.</strong> <code>404 Not Found</code>, <code>401 Unauthorized</code>, <code>403 Forbidden</code>.</li>
    <li><strong>5xx — erro do servidor.</strong> <code>500 Internal Server Error</code>, <code>503 Service Unavailable</code>.</li>
  </ul>
  <p>
    Você vai encontrar esses códigos o tempo todo na aba Network do devtools. Um 404 significa
    que o caminho pedido não existe no servidor; um 500, que o servidor encontrou um erro
    processando a requisição.
  </p>
</LessonSection>

<LessonSection number="05" title="Como o navegador monta a página" id="render-pipeline">
  <p>
    Receber o HTML é só o começo. O navegador ainda precisa transformar esse texto numa página
    visível, e o processo tem etapas bem definidas:
  </p>
  <ol>
    <li>
      <strong>Parse do HTML.</strong> O navegador lê o HTML e monta o DOM
      (<em>Document Object Model</em>), uma árvore em memória que representa a estrutura do
      documento.
    </li>
    <li>
      <strong>Parse do CSS.</strong> Em paralelo, as folhas de estilo são lidas e combinadas em
      outra árvore, o CSSOM (<em>CSS Object Model</em>).
    </li>
    <li>
      <strong>Render tree.</strong> DOM e CSSOM são combinados em uma terceira árvore, só com os
      elementos visíveis e seus estilos aplicados.
    </li>
    <li>
      <strong>Layout.</strong> O navegador calcula a posição e o tamanho de cada elemento
      (também chamado de <em>reflow</em>).
    </li>
    <li>
      <strong>Paint.</strong> Os pixels são desenhados na tela.
    </li>
  </ol>
  <p>
    Quando você edita um elemento com JavaScript (mudar uma cor, mover algo, adicionar um nó), o
    navegador pode precisar refazer partes desse pipeline. Por isso certas mudanças são mais
    caras que outras — assunto que volta no módulo de CSS.
  </p>
</LessonSection>

<LessonSection number="06" title="Cliente vs. servidor" id="cliente-servidor">
  <p>Os termos aparecem o tempo todo e têm uma separação simples:</p>
  <ul>
    <li><strong>Cliente</strong> é quem pede. Na web, o cliente geralmente é o navegador.</li>
    <li>
      <strong>Servidor</strong> é quem responde. Roda em alguma máquina na internet, espera
      requisições HTTP e devolve conteúdo.
    </li>
  </ul>
  <p>
    O HTML, CSS e JavaScript que este curso ensina rodam no cliente, no navegador do usuário —
    por isso são chamados de <em>frontend</em>. Do outro lado da requisição, outras linguagens
    (Node.js, Python, Ruby, Go) rodam no servidor e são chamadas de <em>backend</em>.
  </p>
  <p>
    Frontend e backend não são lados opostos; são camadas complementares separadas pela fronteira
    da requisição HTTP.
  </p>
</LessonSection>

<LessonSection title="Resumo" id="resumo" >
  <ul>
    <li>Uma URL vira um endereço IP através do DNS.</li>
    <li>
      O navegador faz uma requisição HTTP para esse IP e recebe uma resposta com status, headers
      e corpo.
    </li>
    <li>
      O status code resume o que aconteceu — 2xx sucesso, 3xx redirecionamento, 4xx erro do
      cliente, 5xx erro do servidor.
    </li>
    <li>
      O HTML recebido é parseado em DOM, combinado com o CSS em um render tree, posicionado no
      layout e pintado na tela.
    </li>
    <li>
      O que roda no navegador é frontend; o que roda do outro lado é backend. A fronteira é a
      requisição HTTP.
    </li>
  </ul>
</LessonSection>
